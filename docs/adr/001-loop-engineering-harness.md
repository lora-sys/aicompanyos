# ADR-001: Loop Engineering 架构设计决策

## Status
Accepted

## Context

在构建 AI 内容生成系统时，面临一个核心架构选择：是采用单次 LLM 调用的"一次性生成"模式，还是引入多层嵌套循环的迭代优化模式。

**单次调用模式的局限性：**

1. **质量不可控**：单次 LLM 输出质量高度依赖于 prompt 质量，无法在运行时自我修正
2. **缺乏反馈机制**：生成结果无法基于客观标准进行评估和改进
3. **无法积累经验**：每次任务都是独立执行，系统不会从历史任务中学习
4. **评估标准漂移**：如果评估维度不固定，不同轮次的评分不具备可比性

**双层循环的需求来源：**

- **Inner Loop（内层循环）**：Writer → Critic → Writer 的反馈环，用于单次任务的迭代优化。类似人类写作中的"写→审→改"过程。
- **Outer Loop（外层循环）**：全局 Replan 机制，当 Inner Loop 无法收敛到满意结果时触发重新规划。

## Decision

采用 **Planner → Generator → Evaluator + Evolution** 四 Agent 嵌套循环架构：

```
┌─────────────────────────────────────────────────────┐
│                  Outer Loop (Replan)                 │
│  ┌───────────────────────────────────────────────┐  │
│  │            Inner Loop (Writer-Critic)          │  │
│  │                                               │  │
│  │   ┌─────────┐    ┌──────────┐    ┌────────┐  │  │
│  │   │ Planner │ -> │ Generator│ -> │Evaluator│  │  │
│  │   └─────────┘    └──────────┘    └────────┘  │  │
│  │        ^                              |       │  │
│  │        └────────── 反馈 ←─────────────┘       │  │
│  │              （未达标则重写）                    │  │
│  └───────────────────────────────────────────────┘  │
│                     |                               │
│           达到最大迭代次数或平台期                      │
│                     ↓                               │
│               Evolution Agent                        │
│          （提取经验 / 更新能力模型）                    │
└─────────────────────────────────────────────────────┘
```

### 各 Agent 职责

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **Planner** | 任务分解与策略制定 | 用户需求、用户偏好、历史经验 | 执行计划、约束条件 |
| **Generator (Writer)** | 按计划生成内容 | 计划、上下文、风格指南 | 初稿/修订稿 |
| **Evaluator (Critic)** | 基于固定标准评分 | 生成内容、评分标准 | 结构化评分 + 反馈意见 |
| **Evolution** | 从结果中提取经验 | 评分历史、成功/失败模式 | 经验条目、能力更新 |

## Consequences

### 正面影响

1. **Inner Loop（Writer-Critic 反馈环）**
   - 单次任务可进行多轮迭代优化
   - 每次迭代都基于上一轮的具体反馈改进
   - 通过固定评分标准确保反馈的一致性

2. **Outer Loop（全局 Replan）**
   - 当 Inner Loop 连续 N 轮无改善时触发
   - 重新审视原始需求，调整策略方向
   - 避免陷入局部最优解

3. **物理层约束清单**
   - **固定评估标准**：`GradingCriteria` 在任务启动前定义，运行时不可变
   - **退化保护**：若新版本得分低于前一轮最佳分，回滚到最佳版本
   - **平台期检测**：连续 3 轮分数波动 < 5% 时判定为平台期
   - **最大迭代限制**：Inner Loop 上限 10 轮，Outer Loop 上限 3 次 Replan
   - **成本控制**：每轮迭代的 token 消耗有预算上限

### 负面影响

1. **延迟增加**：相比单次调用，多层循环显著增加响应时间
2. **Token 成本上升**：每次迭代都需要额外的 LLM 调用
3. **复杂度提升**：需要管理循环状态、版本历史、收敛条件等

### 风险缓解

- 引入早停机制（early stopping）：当达到目标分数时立即终止
- 缓存中间结果：避免重复计算
- 渐进式策略：初期使用较少迭代次数，根据效果动态调整
