# AI 公司 OS — 北极星

> 这不是路线图。这里不写"Phase 3 做什么"，只写"为什么存在"和"什么时候该往前走"。
> 具体下一步做什么，由跑 v3 之后的真实反馈决定，不由这份文档提前替你决定。

---

## 为什么存在

我一个人用。目的只有一个：**把我反复在做、但其实不需要我亲自动手的工作，变成"写一句 ticket，等结果，看证据"。**

不是为了做一个像公司的东西。"像不像公司"不重要，"我是不是真的少做了事"才重要。

---

## 判断这套系统活得好不好的唯一标准

不是功能多不多，不是架构像不像真公司，是这一句：

> 我现在敢不敢把一个真实想做的任务，写成 ticket 丢进去，然后去做别的事，不盯着它？

只要这句话的答案是"敢"，且交付的东西我看了不用大改，这套系统就是成功的。其余一切（团队、记忆、自动化）都只是为了让这句话的答案更稳，本身不是目的。

---

## 物理焊死（这次重新设计也不变）

1. Critic 反馈完整注入下一轮，从不截断
2. 原始 ticket 原文全程锚定，任何展开/改写不能丢失它
3. 退化保护：最佳版本永不被更差的版本覆盖
4. 最大迭代数永远强制执行，不允许无限跑
5. "完成"必须有证据支撑，能程序化验证的不靠 LLM 自己说

这五条不因为加新功能而妥协。任何新设计如果要求放松这五条中的任何一条，先停下来问为什么，而不是直接改。

---

## 故意不做的事，以及解除条件

这些不是"以后要做的清单"，是"现在主动选择不做，且写明白什么时候才值得重新考虑"：

| 不做的事 | 为什么现在不做 | 什么时候才值得重新考虑 |
|---|---|---|
| CEO / COO 多层 Agent | 只有一个部门时，没有任何跨部门决策需要它们做 | 真实出现两个以上部门，且我发现自己在手动协调它们之间的任务分配时 |
| 动态团队组建（自动分析任务派 Worker） | 我自己判断一个任务该怎么配置，成本几乎是零，自动化它收益不明确 | 我发现自己因为要手动选配置而真的犹豫或出错，而不是"看起来应该有" |
| 多部门、自动认领/抢单 | 单部门的闭环都还没跑顺，调度问题无从谈起 | content 部门跑了足够多真实 ticket，且我真的有第二类任务（如开发）想丢给系统 |
| Web 仪表盘 / 对外服务 | 桌面 UI 本身就是仪表盘，没有第二个人需要远程看 | 出现真实的"需要远程查看进度"的场景，而不是"装着比较酷" |
| 偏好学习模型 | 还不知道简单的记忆回流够不够用 | 验证过记忆蒸馏注入确实提升了产出质量，且仍然觉得不够 |

**这张表的作用**：以后某天我自己或者协助开发的人想加上面任何一项时，先看一眼"解除条件"是否真的满足了，而不是因为"感觉是时候了"或者"这样更像个完整系统"。

---

## 什么时候该往前走一步

不是按时间表，是按这三个信号中的任意一个真实出现：

1. **我亲手卡住了**——某件事我必须手动做，而且重复做了不止一次，自动化它的收益明确大于实现成本
2. **证据告诉我现状不够**——做了 5-10 例真实 ticket 的最小评测，对比某项改动前后的 Critic 分数/我的主观满意度，差距明显
3. **范围被现实推着扩大**——我真的有了第二类、第三类想交给系统做的工作，而不是预判会有

如果以上三条都没出现，按兵不动比加新功能更对。

---

## 这份文档不回答的问题（故意的）

- 不回答"下一步具体加什么功能"——那是 v3 跑起来之后，根据真实反馈才该回答的问题
- 不回答"几个月后系统会长什么样"——会长成什么样，取决于我实际怎么用它，不取决于现在的想象
- 不预设任何完成时间表

如果发现自己在写"Phase X：预计 X 月完成 XX 功能"这样的内容，那已经不是北极星文档，是路线图——路线图不属于这份文件，且历史证明这类提前写好的路线图大概率在写下的两周内就和现实脱节。
