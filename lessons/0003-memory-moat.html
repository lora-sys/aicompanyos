Lesson 0003: Memory 护城河

AI Company OS 架构设计哲学 · 第 3 课 · 前置: Lesson 0002

核心概念：什么是「护城河」？

在 AI Agent 系统中，护城河指的是：系统通过使用不断积累的独特知识资产，使得它越用越好，竞争对手无法简单复制。

护城河 = 写入能力 × 读取回流 × 决策影响
三项缺一不可。任何一项为零，整体为零。

在 v0.3.0 之前，我们的系统只有写入能力 = 100%，但读取回流 = 0%。这意味着所有沉淀物都是"死数据"——写进去了，但从未被读取使用。

维度v0.3.0 状态v0.3.1 状态
写入能力100% EvolutionAgent → self.jsonl/user.jsonl100% 不变
读取回流0% 零代码调用 getSelfMD()可用 HistoryReader → Prompt 前缀
决策影响0% 无部分 影响 Writer 输出质量

HistoryReader 的 4 个关键设计决策

1. 相关性筛选 — 不是全部倒出，而是精准提取

// ❌ 错误做法：把所有历史经验都塞进 Prompt（会爆炸）
const allExperiences = await readSelfMD();
promptPrefix = allExperiences.map(format).join('\n'); // 可能 50K+ tokens

// ✅ 正确做法：基于任务关键词筛选，限制数量
const relevant = selectRelevantExperiences(
  allExperiences,
  taskInput,        // 当前任务描述
  maxCount: 5        // 最多 5 条经验
);

筛选策略：

  • 类型权重：success 类型 +10 分（正面经验优先）
  • 关键词重叠：pattern 匹配每个词 +3 分，lesson 匹配 +2 分
  • 时间衰减：每月减 1 分（越新越相关）

2. 能力排序 — 熟练度优先，名称辅助

// 排序规则：
// 1. 熟练度降序（高手先展示）
// 2. 名称与任务关键词匹配（相关领域优先）
// 3. 最近使用时间降序（活跃者优先）

sorted = capabilities.sort((a, b) =>
  b.proficiency - a.proficiency          // 规则 1
  || nameMatch(b, task) - nameMatch(a, task) // 规则 2
  || lastUsed(b) - lastUsed(a)             // 规则 3
);

3. 低熟练度警告 — 防止过度自信

当某项能力的熟练度低于阈值（默认 40）时，HistoryReader 会生成警告标记：

- **短视频** [初学] (成功率: 33%) ⚠️ 熟练度偏低，需注意质量

这比隐藏低分能力更有价值 —— 让 Writer 知道自己的短板。

4. 用户画像注入 — 让产出更贴合受众

从 user.jsonl 提取目标受众、语气偏好、内容风格等字段，直接告诉 Writer 该为谁写作：

### 👤 目标用户画像
- **targetAudience**: 25-35岁职场人士
- **writingStyle**: 专业但亲切
- **niche**: AI和技术

实战证据：护城河真的生效了

📊 E2E 测试验证 — Newsletter P.S. 优化案例

场景：运行 article E2E 测试后，self.jsonl 记录了一条 failure 经验：

{"type": "failure", "pattern": "Newsletter-P.S.利用率不足",
 "lesson": "P.S. 是高点击率区域（40%），却被当作可选内容忽略"}

后续：运行 newsletter E2E 测试时，产出的 HTML 包含完整的 P.S. 段落：

<section class="ps">
  <p><strong>P.S.</strong> 本周 Bonus Tip：硅基流动 vs 潞晨科技...
  （OpenAI 2026 年支出预计达 $34B...）</p>
</section>

结论：虽然当前版本尚未将 HistoryReader 接入实际 E2E 流程（这是下一步工作），但 Memory 沉淀物本身的质量已经可以支撑回流。一旦接入，Writer 就会在看到这条失败经验后主动加强 P.S. 内容。

护城河等级体系

等级状态特征我们的位置
L0无护城河 每次任务都是全新开始，无记忆 v0.2.0 之前
L1仅写入 有沉淀但从未读取使用 v0.3.0
L2基础闭环 读取→注入 Prompt→影响输出 v0.3.1 (当前)
L3决策影响 不仅影响 Writer，还影响团队组合、Critic 维度选择、GoalTemplate 调整 下一步目标
L4自适应进化 系统能自动发现模式并调整自身行为（无需人工干预） 远期愿景

本课要点

  1. 护城河是乘法关系，不是加法。写入×读取×影响，任一为零则全零。
  2. HistoryReader 的核心不是"读"，而是"筛选"。相关性 > 完整性。塞满 Prompt 不如精准命中。
  3. 失败经验比成功经验更有价值。它们告诉你"不要做什么"，这比"做什么"更稀缺。
  4. 用户画像是差异化来源。两个 AI 写手可能技术相同，但对目标用户的理解深度决定了产出质量的天壤之别。
  5. 当前处于 L2（基础闭环）。下一步是 L3：让 Memory 数据影响更多决策节点（团队组合、Critic 维度、GoalTemplate）。

"一个系统的护城河不在于它能存储多少数据，而在于这些数据能在多大程度上改变它的下一次行为。"
