Lesson 0001: 核心设计哲学
"配置剖面"而非"独立系统"

AI Company OS 的架构核心是一个反直觉的设计决策：

每个部门不是一套独立的系统，而是同一套 Loop Engine 的不同配置剖面（Profile）。

这句话是理解整个系统的钥匙。让我们拆解它。

1. 直觉 vs 现实

✗ 直觉做法：独立系统

为每个部门建独立的代码：

packages/content-production/ → 自己的 Writer, 自己的 Critic packages/rnd/ → 自己的 Writer, 自己的 Critic packages/operations/ → 自己的 Writer, 自己的 Critic

问题：大量重复代码，行为不一致，无法统一进化。

✔ 实际做法：配置剖面

只有一套 Engine，部门只提供配置：

LoopEngine (唯一) → + DeptConfig(article) = 图文生产模式 → + DeptConfig(seed) = 种草生产模式 → + DeptConfig(rnd) = 研发审查模式 (未来)

好处：Engine 进化一次，所有部门受益。

这就是 ADR-005 定义的核心抽象：DepartmentConfig = AgentProfile + GoalTemplate + OutputPipeline + QualityGate

2. 为什么这很重要？三个理由

理由一：复利效应

当 LoopEngine 增加新能力（如更好的 EvolutionAgent、新的 CompletionGuard 策略），所有部门自动获得这个能力，不需要逐个改造。

理由二：快速实验

想尝试一个新的内容格式？只需要写一个新的 DepartmentConfig，不需要改任何 Engine 代码。从想法到验证的时间从"周"缩短到"小时"。

理由三：跨部门知识迁移

内容产出部发现的"高质量 Prompt 模式"，可以通过 Memory 系统沉淀，未来 R&D 部门创建时直接复用。部门的护城河是共享的。

3. 从"静态剖面"到"动态团队"

当前实现是 静态 的：一个 contentType 映射到一组固定的 Agent 配置。

你提出的演进方向是 动态 的：

当前（静态）: contentType("article") → [Writer + Critic] → 永远一样 目标（动态）: taskInput("写一篇需要调研的深度技术文章") → TaskAnalyzer.extractFeatures(task) → { needsResearch: true, complexity: "high", ... } → TeamComposer.compose(features) → [SeniorWriter + Researcher + SeniorCritic] taskInput("写一个小红书种草笔记") → TaskAnalyzer.extractFeatures(task) → { needsResearch: false, hasVisualContent: true, ... } → TeamComposer.compose(features) → [LifestyleWriter + Critic + UIUXDesigner]

关键洞察：团队是任务的函数，不是类型的函数。同一个 "article" 类型，根据任务特征不同，可能组成完全不同的团队。

4. 护城河来自闭环，不是存储

很多系统有"记忆"但没有"护城河"。区别在于：

有存储无护城河有护城河
任务完成 → 写入 self.jsonl 任务完成 → 写入 self.jsonl → 下次任务读取并使用
数据越积越多 系统越来越聪明
数据是"死"的 数据参与决策，是"活"的

AI Company OS 的护城河公式：

护城河 = 写入能力 × 读取回流 × 决策影响

当前状态：写入(100%) × 回流(0%) × 影响(0%) = 0%

MVP 目标：写入(100%) × 回流(100%) × 影响(50%) = 50%

5. 分层职责（记住这张表）

层职责变化频率示例
loop-engine 执行引擎（不变的业务逻辑） 低（月/季） Inner Loop, Outer Loop, CompletionGuard
departments 领域配置（Prompt / 目标 / 输出） 中（周） Writer Prompt, Critic 维度, 平台适配器
team (新) 编排策略（特征→Agent 映射） 高（天） TaskAnalyzer 规则, TeamComposer 策略
knowledge (新) 经验沉淀与复用 持续增长 HistoryReader, CaseRepository

关键要点总结

  1. 部门 = 配置剖面，不是独立系统。这是避免重复和实现复利的根本。
  2. 团队 = 任务的函数，不是类型的函数。同一类型的不同任务可能需要不同的 Agent 组合。
  3. 护城河 = 闭环，不是存储。写了不读 = 没有护城河。
  4. 分层 = 变化频率。变化越快的抽象应该放在越外层（departments > team > knowledge）。
