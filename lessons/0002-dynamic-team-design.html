Lesson 0002: 动态团队
接口设计与规则引擎

基于 Lesson 0001 的哲学（团队 = 任务的函数），本课设计具体的接口和数据结构。

1. 核心接口定义

1.1 IWorker — 团队成员

interface IWorker {
  id: string;                           // 唯一标识
  role: WorkerRole;                     // 角色类型
  agentType: string;                    // 对应 LoopHarness.registerAgent() 的类型
  configOverride?: Partial<WorkerConfig>;  // 此任务的专属配置覆盖
}

type WorkerRole =
  | "writer"        // 内容生成
  | "critic"        // 质量审核
  | "researcher"    // 信息调研
  | "uiux-designer" // 视觉设计
  | "reviewer";     // 最终审查

interface WorkerConfig {
  systemPrompt?: string;   // 覆盖默认 Prompt
  constraints?: Record<string, unknown>;  // 额外约束
  qualityThreshold?: number;  // 该角色的质量门槛
}

设计原则：IWorker 不包含 Agent 实例，只包含描述性配置。实际 Agent 的创建由 LoopHarness 的工厂机制负责。

1.2 ITeam — 一次任务执行的 Agent 集合

interface ITeam {
  id: string;              // 团队 ID (如 "team-abc123")
  taskId: string;          // 关联的任务 ID
  workers: IWorker[];      // 成员列表
  goal: string;            // 团队目标（人类可读）
  features: TaskFeatures;  // 触发此团队组合的任务特征
  createdAt: Date;
}

1.3 TaskFeatures — 任务特征（分析器的输出）

interface TaskFeatures {
  // === 内容特征 ===
  domain: "tech" | "lifestyle" | "finance" | "general";
  needsResearch: boolean;
  hasVisualContent: boolean;
  length: "short" (<500字) | "medium" (500-2000) | "long" (>2000);

  // === 质量特征 ===
  qualityTier: "draft" | "standard" | "premium";

  // === 复杂度指标 ===
  complexity: "low" | "medium" | "high";
  estimatedSteps: number;  // 预估需要几个 Agent Step
}

1.4 ITeamManager — 团队经理（纯编排层）

interface ITeamManager {
  /**
   * 分析任务特征 + 组建团队
   * 这是唯一的公共方法，内部调用 analyzer → composer
   */
  composeTeam(taskInput: string, context: TeamContext): Promise<ITeam>;

  /**
   * 将团队的 Worker 配置转换为 LoopHarness 所需的工厂函数
   * 这是与 LoopHarness 的唯一集成点
   */
  createWorkerFactories(team: ITeam): Map<string, AgentFactory>;
}

interface TeamContext {
  contentType?: ContentType;
  availableRoles: WorkerRole[];  // 该部门支持的可用角色
  departmentId: string;
}

type AgentFactory = (ctx: any) => IGeneratorAgent | IEvaluatorAgent | AgentExecutor;

2. 规则引擎：TaskFeatures → Worker[]

TeamComposer 的核心是一个有序规则表。从上到下匹配，第一个命中的规则决定团队组合。

2.1 规则定义

interface TeamCompositionRule {
  id: string;
  /** 匹配条件（返回 true 表示命中） */
  match: (features: TaskFeatures) => boolean;
  /** 命中后组成的团队 */
  team: Array<{
    role: WorkerRole;
    priority: "essential" | "optional";
    configOverride?: WorkerConfig;
  }>;
  /** 为什么这样组队（用于日志和调试） */
  reasoning: string;
  /** 优先级（数字越小越优先匹配） */
  priority: number;
}

2.2 内容产出部的规则示例

// content-production/src/team/content-rules.ts

const CONTENT_TEAM_RULES: TeamCompositionRule[] = [
  // Rule 1: 高复杂度 + 需要调研 → 全能团队
  {
    id: "research-heavy",
    match: (f) => f.needsResearch && f.complexity === "high",
    team: [
      { role: "writer", priority: "essential", configOverride: { qualityThreshold: 90 } },
      { role: "researcher", priority: "essential" },
      { role: "critic", priority: "essential", configOverride: { qualityThreshold: 85 } },
      { role: "reviewer", priority: "optional" },
    ],
    reasoning: "高复杂度调研型任务：资深写手+调研员+严格审核+可选审查",
    priority: 10,
  },

  // Rule 2: 有视觉内容 → 加入设计师
  {
    id: "visual-content",
    match: (f) => f.hasVisualContent,
    team: [
      { role: "writer", priority: "essential" },
      { role: "critic", priority: "essential" },
      { role: "uiux-designer", priority: "essential" },
    ],
    reasoning: "视觉内容：基础写作+审核+UIUX设计",
    priority: 20,
  },

  // Rule 3: 短内容快速出稿 → 精简团队
  {
    id: "quick-output",
    match: (f) => f.length === "short" && !f.needsResearch,
    team: [
      { role: "writer", priority: "essential", configOverride: { systemPrompt: "快速写手模式" } },
      { role: "critic", priority: "essential", configOverride: { qualityThreshold: 70 } },
    ],
    reasoning: "短内容快速出稿：降低质量门槛以换取速度",
    priority: 30,
  },

  // Rule N: 默认兜底
  {
    id: "default-pair",
    match: () => true,
    team: [
      { role: "writer", priority: "essential" },
      { role: "critic", priority: "essential" },
    ],
    reasoning: "默认 Writer+Critic 配对",
    priority: 999,
  },
];

注意：规则的 priority 决定匹配顺序。更特殊的规则应该有更小的优先级数字（更先匹配）。默认兜底规则优先级最大（最后匹配）。

3. TaskAnalyzer — 特征提取器

TaskAnalyzer 负责只做一件事：从 taskInput 文本中提取结构化的 TaskFeatures。

3.1 实现策略：规则优先，LLM 增强

class TaskAnalyzer {
  analyze(taskInput: string): TaskFeatures {
    const features = this.applyRules(taskInput);
    // 可选：如果置信度低，调用 LLM 做增强分析
    if (features.confidence < 0.7) {
      return this.enhanceWithLLM(taskInput, features);
    }
    return features;
  }

  private applyRules(input: string): TaskFeatures & { confidence: number } {
    return {
      needsResearch: /调研|研究|数据|论文|报告|分析/i.test(input),
      hasVisualContent: /小红书|卡片|配图|设计|视觉|种草/i.test(input),
      length: input.length < 50 ? "short" : input.length < 200 ? "medium" : "long",
      domain: this.detectDomain(input),
      complexity: this.assessComplexity(input),
      qualityTier: "standard",
      estimatedSteps: this.estimateSteps(input),
      confidence: 0.85,  // 规则匹配置信度
    };
  }
}

4. 数据流全景

用户提交任务: "写一篇关于 AI Agent 的深度技术文章，需要调研最新论文" │ ▼ [1] TaskAnalyzer.analyze(taskInput) │ → features = { needsResearch: true, domain: "tech", │ complexity: "high", length: "long", ... } ▼ [2] TeamComposer.compose(features, rules) │ → 按 priority 排序匹配规则 │ → 命中 rule "research-heavy" (priority=10) │ → team = [SeniorWriter, Researcher, SeniorCritic, Reviewer?] ▼ [3] TeamManager.createWorkerFactories(team) │ → Map<"writer", Factory>, Map<"researcher", Factory>, ... ▼ [4] LoopHarness.registerAgent() × N │ → 每个 factory 注册到 Harness ▼ [5] LoopHarness.executeWithLoop(plan) │ → Inner Loop: Writer → Critic (如常) │ → Researcher 通过 Orchestrator 顺序执行 │ → UIUX Designer 通过 Orchestrator 顺序执行 ▼ [6] OutputPipeline → Memory 写入 → 交付

5. 与现有架构的集成点

集成点位置改动量
TeamManager 创建团队 CLI app.ts 或 E2E 测试 +20 行
注册新 Agent 工厂 LoopHarness.registerAgent() 已有 API，无需改动
非 Writer 步骤执行 ExecutionOrchestrator 已有 API，无需改动
部门自定义规则 content-production/src/team/ 新增文件

关键结论：LoopEngine 层面的改动很小。主要工作是：

  1. 在 loop-engine/src/team/ 新增通用接口和基础实现
  2. 在 content-production/src/team/ 新增部门专属规则
  3. 在 CLI/E2E 层接入 TeamManager
